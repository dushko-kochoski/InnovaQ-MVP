# DECISIONS.md — Architectural decision log

## 2026-06-10 — Initial backend build

### D1. Webhook addressing via `route_key` column on Workflow
The constitution's workflow JSON model has no field linking a webhook URL to a
workflow. Added a `route_key` column (UUID, unique, indexed, auto-generated on
create) on the `workflows` table. `POST /v1/triggers/webhook/{route_key}` looks
up the **active** workflow by that key. A UUID key is unguessable, so the
endpoint can stay unauthenticated (external systems posting webhooks won't have
our JWT cookies). Paused/draft workflows return 404 so triggers can't fire them.

### D2. Steps stored as a JSON column, validated at the boundaries
`Workflow.steps` and `WorkflowTemplate.steps` are SQLAlchemy `JSON` columns
(portable across SQLite and PostgreSQL). Validation happens twice: Pydantic
schemas (`WorkflowStep` + `validate_steps_order`) reject bad steps at the API
boundary, and `workflow_parser.parse_steps` re-validates at execution time so a
row edited outside the API can never crash the engine — it raises
`WorkflowParseError`, which the webhook background task logs and swallows.

### D3. Step semantics: ordered, first step is a trigger, false condition halts
Steps execute in ascending `step` number. A workflow must start with a trigger
step; trigger steps are no-ops at execution time (the trigger already fired).
A condition evaluating false stops the run — there is no branching in v1.
A missing payload field or operator type mismatch evaluates to **false**
(workflow quietly does not proceed) rather than raising; an unreachable
condition shouldn't 500 a customer's webhook pipeline.

### D4. `viber` and `email` actions are logging stubs
No Viber/SMTP credentials exist yet. These action types log the intended send
and return `{"ok": true}` so workflows are executable and observable
end-to-end today. `http_request` is fully implemented via httpx with a
configurable timeout. Real channel integrations are a later milestone — the
`execute_action` dispatch is the single place to add them.

### D5. JWT in httpOnly cookie, not Authorization header
Per the constitution. Cookie name lives in config (`AUTH_COOKIE_NAME`).
`secure` flag is `not DEBUG` so local http dev works while production
(DEBUG=false) requires https. SameSite=lax. Logout simply deletes the cookie —
no server-side token blacklist in v1 (24h expiry bounds the risk).

### D6. Auth failures are uniform 401/404
Login returns the same 401 for unknown email and wrong password (no account
enumeration). Workflow routes return 404 — not 403 — when a workflow exists
but belongs to another user, so IDs can't be probed for existence.

### D7. Seeding is idempotent and runs on startup
`seed.py` upserts the 15 templates keyed by a stable `slug` column (added
beyond the constitution's model — names/descriptions can change, slugs can't).
`main.py` runs `create_all()` + seed in the FastAPI lifespan, so a fresh clone
boots with a ready database. `create_all` is sufficient pre-production;
Alembic migrations are deferred until the schema stabilizes.

### D8. PUT /v1/workflows/{id} is partial-update
All fields on `WorkflowUpdateRequest` are optional; only provided fields are
applied. Strictly PUT-as-replace would force clients to resend full step
arrays to toggle `active`/`paused`, which the dashboard does constantly.

### D9. Test isolation via dependency override + in-memory SQLite
Tests override the `get_db` dependency with an in-memory SQLite engine
(`StaticPool` so all threads share one DB) and skip the app lifespan, so the
suite never touches `innovaq.db`. httpx `ASGITransport` runs FastAPI
BackgroundTasks before the response context closes, which lets
`test_webhooks.py` assert the engine was invoked with the right payload.

### D10. UUIDs stored as String(36)
SQLite has no native UUID type. String PKs generated by `uuid.uuid4()` work
identically on SQLite and PostgreSQL; can be migrated to native `UUID` columns
when PostgreSQL becomes the only target.

### D11. APScheduler installed but not yet wired
`schedule` and `http_poll` trigger types are accepted and stored, but only
`webhook_receive` actually fires in v1. APScheduler is in requirements for the
scheduled-trigger milestone; wiring it into the lifespan is deliberately
deferred to keep this build focused on the webhook path.

### D12. Startup "errors" were environment mismatch, not code bugs
A reported `ModuleNotFoundError: No module named 'sqlalchemy'` on startup was
reproduced only with the **global** Python: on this machine, bare `pip`
belongs to Python 3.14 while `python` is 3.11, so `pip install` lands packages
where `python` never sees them. The project venv (`.venv`) has all
dependencies, and `uvicorn backend.main:app --reload` from the project root
starts cleanly (uvicorn puts the cwd on `sys.path`, so the `backend.*` imports
need no changes). Resolution: README setup now mandates `python -m pip` /
`python -m uvicorn` inside the activated venv and includes a troubleshooting
table. No source changes were required.

### D13. Execution log added for the dashboard's "recent triggers" widget
The dashboard needs a recent-webhook-triggers log, but v1 never persisted
runs. Added `WorkflowExecution` (workflow_id, status success|error, results
JSON, triggered_at), written by the webhook background task. The task opens
its own session via module-level `SessionLocal` — the request session is
already closed when background tasks run (FastAPI ≥0.106 tears down yield
dependencies before tasks), and the module-level symbol lets tests patch it
to the in-memory engine. Recording failures are logged and swallowed: an
execution must never fail because its bookkeeping did. Exposed as
`GET /v1/triggers/recent?limit=` (auth-gated, joins through Workflow to scope
to the current user).

### D14. App pages auth via API probe; frontend must be served same-site
The JWT cookie is httpOnly, so the app pages cannot inspect it from JS. The
"check for JWT cookie on load" is implemented as a credentialed
`GET /v1/workflows`; a 401 redirects to `/index.html`. Because the cookie is
`SameSite=Lax`, browsers only attach it when frontend and API are same-site:
serve the frontend from `http://127.0.0.1:3000` when the API is on
`http://127.0.0.1:8000` (hostnames must match; ports don't matter for
SameSite). `http://127.0.0.1:3000` was added to the CORS allowlist alongside
`localhost:3000` for this reason.

### D15. Railway deployment via Nixpacks; postgres:// URL normalization
Deployment targets Railway with the stock Nixpacks Python builder —
`Procfile` + `railway.json` (start command, `/health` healthcheck, restart
policy), no Dockerfile to maintain. Railway's PostgreSQL plugin hands out
`postgres://` URLs, which SQLAlchemy 2.x rejects; a `field_validator` on
`Settings.DATABASE_URL` rewrites the scheme to `postgresql://`, so the
Railway variable can be referenced as-is. `psycopg2-binary` added to
requirements (fine at this scale; switch to source-built `psycopg2` only if
wheels become a problem). Startup still runs `create_all()` + seed, which is
idempotent on PostgreSQL too — Alembic remains deferred (see D7).

### D16. Production cookies are SameSite=None; CORS driven by FRONTEND_URL
In production the frontend (Netlify) and API (Railway) are different sites,
so the `SameSite=Lax` dev cookie would never be attached to fetch calls.
`DEBUG` now switches the auth cookie: `false` → `SameSite=None; Secure`
(cross-site, https only), `true` → `SameSite=Lax` without `Secure` (local
http, same-host setup per D14). CORS allowlist is `{FRONTEND_URL}` from the
environment, with the two localhost:3000 spellings appended only in DEBUG —
production never exposes dev origins. FRONTEND_URL is normalized to strip a
trailing slash, since CORS origin matching is exact.

### D17. Frontend API base URL via config.js, not per-page constants
All five API-calling pages load `/config.js` before their inline script and
read `window.API_BASE_URL` (classic script + global, not an ES module — no
build step, works from file drag-and-drop hosting). `config.js` picks the
local API when served from `localhost`/`127.0.0.1` and `PRODUCTION_API`
otherwise, so the same artifact works in dev and on Netlify; the production
URL is edited in exactly one place after the backend deploy.

### D18. i18n via text-matching runtime, not per-element attributes
The МК/EN toggle (default: Macedonian, persisted in localStorage under
`innovaq_lang`) is implemented in `frontend/translations.js`, which holds
both the 300-key en/mk dictionary and the runtime. Instead of hand-tagging
every element with `data-i18n` across 14 pages, the runtime annotates
elements at load time by matching their normalized English text (tags
stripped, `&amp;` decoded, whitespace collapsed) against the `en` map, then
swaps `innerHTML` on toggle; placeholders match `el.placeholder` the same
way. Pages therefore only carry the script include and the toggle markup
(injected by `scripts/inject_i18n.py`). The trade-off — a dictionary value
that drifts from the page text silently stays English — is policed by
`scripts/check_i18n.py`, which fails if any static `en` value matches no
page (run it after editing copy). Dynamic JS-rendered strings use
`window.t("js.*")` and pages re-render on the `langchange` event. DB-stored
template names/descriptions stay as seeded (English) — translating data is
a backend concern for later. Cyrillic plan names (Старт/Бизнис/Про) are
identical in both languages by design.
